## 1. 웹사이트 첫 화면이 3배 빨라진 이유 (2026-07-17)

### 결과 먼저

| 지표 | 고치기 전 | 고친 후 |
|---|---|---|
| 성능 점수 (Lighthouse) | 43점 | 82점 |
| 첫 화면 표시 (FCP) | 9.3초 | 2.8초 |
| 화면 출렁임 (CLS) | 0.259 | 0.024 |

코드를 한 줄도 "빠르게" 만들지 않았다. **순서만 바꿨다.**

### 용어 3개만 알면 됨

- **FCP** (First Contentful Paint): 흰 화면에서 **뭐라도 처음 보이기까지** 걸리는 시간. 사용자가 "느리다"고 느끼는 그 시간.
- **LCP** (Largest Contentful Paint): 화면에서 **제일 큰 덩어리**(제목, 큰 이미지)가 보이기까지의 시간. 구글이 검색 순위에 반영하는 지표.
- **CLS** (Cumulative Layout Shift): 읽으려는 순간 **화면이 출렁여서 버튼 위치가 바뀌는 정도**. 0에 가까울수록 좋음.

### 무엇이 문제였나 — "렌더 차단"

브라우저는 HTML을 읽다가 `<link rel="stylesheet">`를 만나면 **그 CSS를 다 받을 때까지 화면 그리기를 멈춘다.**
스타일 없이 글자를 먼저 보여줬다가 CSS 도착 후 다시 그리면 화면이 번쩍이기 때문에, 일부러 기다리는 것.
이걸 **렌더 차단(render-blocking)**이라고 한다.

우리 페이지의 `<head>`에는 이게 있었다:

```html
<link href="https://fonts.googleapis.com/css2?family=Jua&..." rel="stylesheet">
```

구글 폰트 CSS(25KB)를 **다 받기 전까지 화면이 하얗다.** 게다가:

1. 우리 서버(Cloudflare)가 아니라 **구글 서버로 새로 접속**해야 함 (DNS 조회 + 연결 수립 = 왕복 몇 번)
2. 한글 폰트는 글자가 11,172자라 **영문 폰트보다 수십 배 무거움** (구글이 100여 조각으로 쪼개 주지만, 첫 화면에도 여러 조각 필요)
3. 느린 4G(Lighthouse 모바일 기준)에서는 이 대기가 **수 초**가 됨

→ 그래서 첫 화면이 9초였다. 내용은 0.1초면 그릴 수 있는데 **폰트를 기다리느라.**

### 어떻게 고쳤나 — "비동기 CSS 로드" 패턴

```html
<!-- 전 -->
<link href="...폰트CSS..." rel="stylesheet">

<!-- 후 -->
<link href="...폰트CSS..." rel="stylesheet" media="print" onload="this.media='all'">
<noscript><link href="...폰트CSS..." rel="stylesheet"></noscript>
```

동작 원리:

1. `media="print"` — "이 CSS는 **인쇄할 때만** 필요해"라고 선언. 브라우저는 인쇄용 CSS를 위해 화면 그리기를 멈추지 않는다. → **차단 해제, 다운로드는 뒤에서 계속**
2. `onload="this.media='all'"` — 다운로드가 끝나는 순간 "사실 화면용이야"로 바꿔치기. → 폰트 적용
3. `<noscript>` — 자바스크립트를 꺼둔 브라우저를 위한 보험 (원래 방식으로 로드)

결과: 화면은 **일단 기기 기본 글꼴로 즉시** 그려지고(2.8초), 폰트가 도착하면 Jua체로 갈아입는다.

### CLS(출렁임)는 왜 같이 좋아졌나

전에는 "흰 화면 9초 → 폰트까지 입은 완성 화면"이 한 번에 나타나는 게 아니라,
중간에 요소들이 자리를 잡으며 크게 출렁였다 (0.259).
지금은 기본 글꼴로 먼저 자리를 잡고 폰트만 갈아입으니 이동량이 작다 (0.024).

### 교훈

- **속도 최적화의 첫 질문은 "무엇이 화면 그리기를 막고 있나"** — 코드가 느린 경우보다 **기다리는 순서가 나쁜** 경우가 훨씬 많다.
- 측정 없이 고치지 말 것. Lighthouse 진단 → 원인 특정(렌더 차단 폰트 CSS) → 그것만 수리 → 재측정. 이번에 건드린 건 사실상 link 태그 한 줄이다.
- 한글 웹폰트는 항상 성능 용의자 1순위. 쓰되, 렌더를 막게 두지 말 것.

### 같이 알아두면 좋은 것 (이번에 이미 적용돼 있던 것들)

- `<link rel="preconnect" href="https://fonts.gstatic.com">` — "곧 이 서버 쓸 거니까 미리 악수해 둬". 연결 수립 시간을 선불로 냄.
- `&display=swap` (폰트 URL 안) — "폰트 늦으면 기본 글꼴로 먼저 보여줘". 이번 수리는 이걸 CSS 파일 자체로 확장한 셈.
- GA·애드센스 스크립트의 `async` — 스크립트도 CSS처럼 렌더를 막을 수 있어서, 분석·광고 코드는 항상 async로.

## 2. "동작을 하나도 안 바꾸는 수술" — 리팩토링과 검증 (2026-07-17)

### 무슨 작업이었나

한글 따라쓰기 페이지(hangul/index.html)는 608줄짜리 파일 하나에
**화면 코드**(버튼, 미리보기 그리기)와 **계산 코드**(글자를 몇 칸에 어떻게 배치할지)가 섞여 있었다.
이 중 계산 코드만 `hangul-gen.js`라는 별도 파일로 옮겼다. 겉보기엔 아무것도 안 변했다 — 그게 목표다.

### 왜 하나 — 리팩토링의 이유

**리팩토링(refactoring)** = 동작은 그대로 두고 코드의 구조만 고치는 것.
"잘 돌아가는데 왜 건드려?"에 대한 답은 두 가지다:

1. **테스트가 가능해진다.** 화면 코드는 브라우저가 있어야 돌지만, 순수 계산 코드는
   터미널에서 `node`로 1초 만에 검사할 수 있다. 분리 전엔 한글 도구에 자동 테스트가 0개였고,
   분리 후 24개가 생겼다. 이제 누가(사람이든 AI든) 이 로직을 잘못 고치면 **기계가 잡아낸다.**
2. **다음 수정이 싸진다.** 계산 규칙을 고칠 때 화면 코드를 헤집을 필요가 없고, 그 반대도 마찬가지.

### 핵심 개념 — 순수 함수 (pure function)

**같은 입력 → 항상 같은 출력, 그리고 바깥세상(화면, 파일, 네트워크)을 안 건드리는 함수.**

```
buildRowsChar("가나다", "mid", 3)  →  항상 똑같은 줄 배열
```

순수 함수는 (1) 테스트하기 쉽고 (2) 어디서든(브라우저·서버·터미널) 돌고 (3) 버그가 나면 원인이 그 안에만 있다.
글씨방의 미로(maze-gen.js), 수학(generators.js), 한글(hangul-gen.js)이 전부 이 패턴이다.
반대로 "화면에 그리기"는 순수할 수 없으니 페이지 쪽에 남긴다. — **계산과 그리기의 분리.**

### 진짜 중요한 부분 — "안 바뀌었음"을 어떻게 증명하나

리팩토링의 최대 위험은 "옮기다가 미묘하게 달라지는 것"이다. 눈으로 비교하는 건 못 믿는다. 그래서:

1. **고치기 전에** 대표 상태 8가지(중복 글자, 짧은 이름, 긴 단어, 여러 장, 빈 입력…)로
   페이지가 만든 결과물(HTML)을 통째로 저장해 둔다 — 이게 "정답지"
2. 리팩토링한다
3. **같은 8가지를 다시 뽑아서** 정답지와 **바이트 단위로 비교**한다
4. 8/8 완전 일치 → 동작이 안 바뀌었다는 **증명**이 된다

이번에 실제로 이 절차를 썼고, 미로 리팩토링 때도 같은 방법을 썼다 (그땐 "옛 공유 링크가 같은 미로를 재현하는가"까지 검증).

### 곁다리 교훈 — 캡처 스크립트의 버그

정답지를 뽑는 스크립트가 처음에 8개 상태를 전부 "1장"으로 기록했다.
원인: 주소에서 `#해시`만 바뀌면 브라우저는 **페이지를 다시 로드하지 않는다.**
그래서 첫 상태만 8번 캡처하고 있었던 것. `about:blank`를 한 번 거쳐가게 해서 해결.
→ **검증 도구 자체도 검증해야 한다.** "전부 통과"가 너무 쉽게 나오면 의심할 것.
(이번엔 "여러 장짜리 상태가 왜 1장이지?"라는 위화감이 버그를 잡았다.)

### 교훈

- 리팩토링은 기능 추가와 **절대 같이 하지 않는다.** 같이 하면 "달라진 게 리팩토링 때문인지 기능 때문인지" 구분이 불가능해진다.
- 순서는 항상: **정답지 확보 → 수술 → 대조 → 테스트를 유산으로 남김.** 테스트가 남으니 다음 사람(미래의 나)이 보호받는다.
- 파일이 길다는 것 자체는 죄가 아니다. **"계산"과 "화면"이 섞여 있는 게** 문제다. 분리 기준은 줄 수가 아니라 책임.

## 3. "테스트가 흔들리면 환경부터 의심하라" — 구구단 리팩토링에서 잡은 유령 (2026-07-17)

### 상황

구구단 도구도 한글과 같은 방법(정답지 캡처 → 분리 → 대조)으로 리팩토링했다.
문제 순서가 랜덤이라, 비교가 가능하도록 **Math.random을 시드 고정**(같은 시작값 → 항상 같은 난수열)해서 캡처했다.
그런데 4개 캡처 중 1개가 리팩토링 전후로 달랐다 — 5×3이 나와야 할 자리에 2×6이.

### 범인은 코드가 아니라 환경

캡처 페이지에는 구글 애널리틱스(gtag.js)가 같이 로드되는데, **이 외부 스크립트도 내부에서 Math.random을 쓴다.**
난수열은 하나로 이어져 있어서, gtag가 난수를 **몇 번 꺼내 쓰느냐에 따라** 우리 셔플이 받는 난수가 밀린다.
그리고 그 횟수는 네트워크 타이밍에 따라 매번 달랐다. 즉 **리팩토링과 무관한 불일치**였다.

해결: 캡처 브라우저에서 **외부 요청을 전부 빈 응답으로 차단**(밀폐된 환경) → git에 저장된 리팩토링 전 코드와
현재 코드를 같은 조건에서 재캡처 → 4/4 바이트 동일. 이제야 진짜 증명이 됐다.

### 교훈

- **비결정성(같은 걸 돌려도 결과가 달라짐)의 3대 용의자: 랜덤, 시간, 네트워크.** 재현 가능한 테스트를 원하면 셋 다 통제해야 한다. (스모크 테스트도 같은 이유로 외부 요청을 차단하고 돈다)
- 테스트가 실패하면 "코드가 틀렸다"와 "테스트가 틀렸다" 두 가설을 **같은 무게로** 조사할 것. 이번엔 후자였다.
- 리팩토링 전 코드는 지워진 게 아니다 — **git에 있다.** `git archive HEAD`로 옛 버전을 통째로 꺼내 현재와 나란히 실행·비교할 수 있다. 커밋을 잘게 하는 이유 중 하나.

### 덤 — 이번 분리로 얻은 것

구구단 시험의 문제 생성·채점·사지선다 로직이 `gugudan-gen.js`(38줄 순수 모듈)로 나왔고,
이로써 **미로·수학·한글·구구단 4대 생성 로직이 전부 "순수 모듈 + 자동 테스트" 구조**가 됐다.
남은 인라인 코드는 화면 조작뿐이라, 다음 기능(예: 나눗셈 시험)은 모듈에 함수 하나 얹는 일이 된다.

## 4. 낱말 미로 (2026-07-23) — "성질"을 테스트하면 2,000개를 공짜로 검사한다

### 결과

한글 낱말 미로(/maze/hangul/)의 핵심 약속은 하나다: **"정답 길을 따라가면 글자가 반드시 낱말 순서로 모인다."**
이걸 낱말 10종 × 시드 100개 × 난이도 2종 = **2,000가지 조합으로 자동 검사했고 전부 통과**한 뒤에야 배포했다.

### 무엇을 했나

새 기능을 테스트하는 방법은 두 가지다.

1. **예시 검사** — "'사과'를 넣으면 이 미로가 나온다"처럼 정답을 몇 개 박아두고 비교.
2. **성질 검사** — 어떤 입력이든 항상 참이어야 하는 **성질(불변식)**을 정하고, 입력을 대량으로 부어 넣기.

낱말 미로는 랜덤 생성이라 1번이 어렵다(정답이 매번 다르니까). 대신 성질은 명확하다:

- 길 위의 글자를 순서대로 이으면 = 입력한 낱말
- 함정 글자는 절대 정답 길 위에 없다
- 함정 글자에 낱말의 글자가 섞이지 않는다
- 같은 시드면 항상 같은 미로 (공유 링크 재현)

이 4가지를 코드로 적어두면, 검사 입력은 반복문이 공짜로 만들어 준다.

### 교훈

- **랜덤 생성 기능은 "정답 비교"가 아니라 "성질 검사"로 테스트한다.** 성질 4줄 + 반복문 = 사람이 평생 못 해볼 2,000판 검증.
- 성질을 문장으로 못 적겠다면 기능 설계가 아직 덜 된 것이다. "이 기능이 절대 어기면 안 되는 약속이 뭐지?"가 먼저.
- 기존 코드를 건드릴 땐(이번엔 미로 SVG에 글자 얹기) **"옵션이 없으면 출력이 한 글자도 안 변한다"** 를 별도 성질로 검사 — 신기능이 구기능을 몰래 깨는 걸 막는 안전핀.
