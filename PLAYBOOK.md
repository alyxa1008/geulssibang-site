# 웹 서비스 출시 플레이북 — 글씨방으로 배운 것

아이디어 → 라이브 서비스까지 실제로 밟은 순서. 다음 프로젝트에서 그대로 반복할 것.

## 0단계. 기획 — 만들기 전에 정할 것

1. **실생활 불편에서 출발** — "받아쓰기를 매번 불러줘야 한다" 같은 구체적 고통
2. **경쟁 조사 먼저** — 검색해서 기존 강자 확인. 강자가 있는 영역(수학=일일수학)은 보조로,
   빈 영역(받아쓰기 불러주기)을 간판으로. 돈 쓰기 전에 경쟁력 검증
3. **기술은 최소로** — 빌드 없는 정적 HTML/CSS/JS. 서버·DB·프레임워크 없음 = 유지비 0, 고장 0

## 1단계. 제품 — 만들 때 지킨 원칙

- **코드 구조**: 공통(assets/style.css, common.js) + 도구별 폴더 하나씩.
  데이터와 화면 분리(예: math/generators.js 등록부 — 유형 추가가 객체 하나 추가로 끝남)
- **실물 벤치마킹** — 시판 학습지를 보고 "격자가 페이지를 꽉 채운다" 같은 디테일을 따름
- **사실 검증** — 교육과정 같은 도메인 지식은 감이 아니라 조사로 (학년 분류를 검색으로 확인)
- **자동 검증 습관** — 생성 문제 수만 개 자동 채점, 화면 조합 시뮬레이션. 눈검사보다 빠르고 확실
- **주기적 코드 정리** — 기능 몇 개 추가할 때마다: 죽은 코드, 복붙 중복(공통화), 메모리 누수 점검

## 2단계. 출시 인프라 — 이 순서 그대로 (순서가 돈과 시간을 아낌)

```
1. git init → GitHub 저장소 → push          (백업·이력·복구)
2. Cloudflare Pages 배포                     (임시주소 *.pages.dev로 즉시 라이브)
3. 도메인 구매 (Cloudflare Registrar, .com)  (애드센스는 본인 도메인 필수)
4. DNS CNAME 연결 + 사이트 내부 주소 일괄 치환 (canonical/og/sitemap/robots)
5. 파비콘 + OG 이미지                        (카톡 공유 미리보기)
6. GA4 설치 + 전환 이벤트 + JS 에러 모니터링  (체류시간보다 "인쇄 클릭" 같은 전환을 측정)
7. 구글 서치콘솔 + 네이버 서치어드바이저 등록
   → 사이트맵 제출 → 핵심 페이지 수동 색인 요청 (색인이 하루 만에 시작됨)
```

- **도메인을 검색 등록 전에** 사야 SEO를 두 번 안 함
- 평소 업데이트 루프: `수정 → 테스트 → commit/push → wrangler deploy → 라이브 검증`
  (배포가 조용히 실패한 적 있음 — 배포 후 curl로 실제 반영 확인 필수)

## 3단계. 검색(SEO) — 기술이 아니라 단어 싸움

- **제목 = 사람들이 실제 치는 검색어** ("생성기" ✗ → "프린트", "출력", "무료" ✓)
- **도구마다 콘텐츠 페이지 1개씩 짝** — 도구는 "쓰러" 오고 콘텐츠는 "검색으로" 옴
  (받아쓰기↔급수표, 한글↔떼기 가이드, 수학↔학년 로드맵)
  콘텐츠에서 클릭 한 번으로 도구가 채워져 열리는 연결이 핵심 (공유 링크 형식 재활용)
- 페이지마다: title/description/canonical/og + FAQ + JSON-LD, 내부 링크, 404 페이지
- 흰 카드 안 글자색 같은 시각 회귀는 공용 클래스로 한 곳에서 관리

## 4단계. 유입과 운영 — 만들기가 끝나면 측정-투자 루프

- **블로그 먼저** (내 자산, 제약 없음, 네이버 검색에 잡힘) → **카페는 나중에**
  (등업 필요, 홍보 엄격 — 규정 확인 → 운영진 문의 → 지역맘부터. 강퇴되면 채널 영구 손실)
- **주기 체크 계기판 4개**: GA4(주 1~2회: 유입 출처·이벤트), 서치콘솔(주 1회: 노출 검색어·색인),
  네이버 어드바이저(주 1회), Cloudflare(월 1회)
- **운영 루프**: 데이터 확인 → 반응 있는 키워드/도구 발견 → 그쪽 보강 → 다시 확인.
  운이 아니라 반복. 초기 GA4의 Direct=본인, 미국 0초=봇 — 노이즈 걸러 읽기
- **애드센스**: 구조 요건(개인정보처리방침의 쿠키·광고 조항, 소개·연락처, 독창 콘텐츠, 깨진 링크 0)
  + 시기 요건(색인 완료 + 트래픽). 구조만 되면 신청하지 말고 색인·방문자 생긴 뒤 신청

## 교훈 한 줄씩

- 만들기 45% / 검증 15% / 인프라 15% / 검색·콘텐츠 15% / 운영 준비 10% — 코딩은 절반이다
- 모든 단계에서 "자동으로 확인할 수 있는가?"를 먼저 물을 것
- 데이터 장비(GA4·서치콘솔)는 손님이 오기 전에 깔아야 첫 손님부터 보인다
- 순서를 지키면 같은 일을 두 번 안 한다 (도메인→치환→등록)
